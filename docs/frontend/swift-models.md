# Data Models

All persistent models use SwiftData. Photos are stored as files in the app's Documents directory — models hold file path references only.

## UserProfile

Stores the user's onboarding results. Singleton — only one per app.

```swift
UserProfile
├── id: UUID
├── createdAt: Date
├── updatedAt: Date
│
│  // Onboarding results (collected via Guided UI)
├── interests: [String]                       — things they're curious about
├── growthAreas: [String]                     — where they want to grow
├── vibe: [QuestVibe]                         — how they want to experience quests (e.g., Solo, Social, Adventurous, etc)
├── experimentationLevel: Int                 — "Spontaneity" (1-5 slider) for wildcard quests
├── budget: [BudgetLevel]                     — .free, .cheap, .moderate, .splurge
├── transportation: [TransportationMode]      - .walking, .publicTransport, .car, .bike, .rideshare
├── locationPreferences: [LocationPreference] — preferred environments (e.g., Nature, Indoors)
├── additionalContext: String?                — optional nuance text
│
│  // Location
├── city: String                       — user's base city (e.g., "San Francisco")
├── cityLatitude: Double?              — latitude of the city center (for location-based quest generation)
├── cityLongitude: Double?             — longitude of the city center (for location-based quest generation)
│  // State
└── hasCompletedOnboarding: Bool
```

## Sidequest

The core model. Represents a single sidequest regardless of its state.

```swift
Sidequest
├── id: UUID
├── createdAt: Date
│
│  // Content (from Cloud Function response)
├── title: String
├── questDescription: String
├── difficulty: DifficultyRating       — .easy, .moderate, .hard, .extreme
├── estimatedActivityMinutes: Int      — estimated activity duration in minutes
├── categories: [String]               — classification labels (e.g., adventure, creativity)
│
│  // Location (optional — nil if quest is not location-based)
└── locationInformation: LocationInformation?
│
│  // State
├── status: SidequestStatus            — .available, .skipped, .active, .completed
├── completedAt: Date?
│
│  // Get Started guide (generated on demand, cached)
├── getStartedSteps: [String]?         — nil until user requests it
│
│  // Completion data
├── journalEntry: String?              — user's reflection text
└── journalPhotoFilenames: [String]    — references to photos in Documents dir
```

### LocationInformation

Codable struct grouping all location data. Present only on location-based quests.

```swift
LocationInformation
├── name: String                       — name of the place (e.g., "Dolores Park")
├── address: String                    — full address string
├── locationDescription: String        — short 1-2 sentence place summary generated by the backend LLM (not from Google Maps; named to avoid clashing with NSObject's `description`)
├── latitude: Double                   — for MapKit display
├── longitude: Double                  — for MapKit display
├── photoURL: String                   — photo URL from Google Maps Places API
├── googleMapsURL: String              — URL to open the place in Google Maps
├── distanceMiles: Double?             — calculated client-side from cityLatitude/cityLongitude (not from backend)
└── transportationOptions: [TransportationOption] — computed client-side (not from backend)
```

### TransportationOption

```swift
TransportationOption
├── mode: TransportationMode           — .walking, .car, .publicTransport, etc.
├── estimatedTravelMinutes: Int        — estimated travel time in minutes
└── isRecommended: Bool                — true if AI recommends this mode
```

## Batch Lifecycle

Sidequests are generated in **batches of 10**. Here's how the batch flows through the system:

```swift
Generate batch (10 quests, all .available)
    │
    ├── User swipes left → quest becomes .skipped
    ├── User swipes right → quest becomes .active (1 at a time)
    │
    ├── User wants to swap → .active quest returns to .available, pick another
    │
    ├── All 10 skipped → show list view, user must pick one
    │
    └── User completes active quest → quest becomes .completed
        │
        ├── Remaining quests stay in batch (.available / .skipped)
        ├── User can pick next quest from remaining batch
        │
        └── "Generate new batch" option appears in Discover tab
            │
            ├── User chooses to regenerate → remaining quests deleted, new 10 generated
            └── User chooses to keep picking → batch continues shrinking until empty
                │
                └── When batch is fully exhausted (all completed or 0 remaining)
                    └── Auto-generate new batch of 10
```

**Key rules:**

- Only **one quest can be .active** at any time
- A batch **persists across completions** — completing a quest does not auto-clear the batch
- The user can keep picking from remaining quests or explicitly regenerate for a fresh batch of 10
- Regenerating clears all remaining `.available` and `.skipped` quests from SwiftData
- When the batch is fully exhausted (no `.available` or `.skipped` quests remain), a new batch is auto-generated

## Onboarding

The onboarding is a paginated Guided UI flow — multi-step forms with selectable pills, sliders, and minimal typing. No chat interface.

- The app preserves the form state locally (e.g., via AppStorage or a draft model) as the user progresses through the steps.
- If the user quits mid-onboarding, their progress is saved and they resume where they left off next time.
- Once the user completes all steps, the final `UserProfile` is fully populated and saved in SwiftData.
- The app then calls `generateCuratedSidequests` passing the full, structured `UserProfile` object to the backend (the batch count is server-controlled; the client does not send a count).
- The backend may return a pre-generated batch instantly (cache hit) or generate fresh (the app shows a "curating" loading state while waiting). No per-user rate limit is enforced on the backend during the testing phase; any usage limiting will be added client-side.
- A separate `generateUserDescribedSidequest` endpoint takes a freeform `prompt` + `UserProfile` and returns one tailored sidequest (also no backend cap for now). The iOS client will need the same rename from the old `generateSidequests`.

## Relationships

```swift
UserProfile (singleton — only one per app)
    │
    └── has many → Sidequest (all quests ever generated)
                      │
                      └── has many → photo files (via journalPhotoFilenames)
```

There's effectively one UserProfile. All Sidequests belong to that profile. The relationship is implicit (no explicit SwiftData relationship needed since there's only one user).

## Notes

- **No separate JournalEntry model** — journal text lives directly on the Sidequest. Since it's one entry per quest, a separate model adds complexity without benefit.
- **One active quest at a time** — the user picks one sidequest from a batch of 10 via the swipe interface. They can swap their active quest for a different one from the same batch, but only one can be `.active` at any given time.
- **Batch of 10** — the backend generates 10 quests per batch. The batch persists until the user explicitly regenerates or exhausts all remaining quests. This naturally limits backend costs.
- **No delete, only swap** — users can't delete sidequests from their batch. They can skip (swipe left) or swap their active quest for a different one from the batch.
- **Photos are stored as filenames, not full paths** — the Documents directory path can change between app launches. Store just the filename and resolve the full path at runtime.
